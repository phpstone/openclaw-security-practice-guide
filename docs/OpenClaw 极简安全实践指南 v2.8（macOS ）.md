

 │ 适用场景：OpenClaw 运行于 macOS，运行用户不是系统管理员；可按需授予有限 sudo
 │ 白名单权限；仍是多用户运行场景；系统隐私与设备访问由 macOS TCC 负责控制。
 │ 核心原则：日常低摩擦，高危必确认，持续显性化巡检，拥 抱零信任（Zero Trust）。

 │ TCC 授权或系统级敏感权限变更，必须提交人工确认与人工操作。

 4. TCC 权限处理机制（本修订版核心差异）

 ### 4.1 原则

 macOS 隐私权限由 TCC 控制。
 Agent 只能：

 - 检测权限是否缺失
 - 向人工说明缺什么权限
 - 告知为什么需要
 - 告知去哪里授权
 - 授权后再重试

 Agent 不得：

 - 修改 TCC 数据库
 - 伪造授权状态
 - 自动批准系统隐私权限
 - 以技术手段绕过隐私弹窗

 ### 4.2 标准处理流程

 当任务依赖以下能力时，应进入 TCC 检查流程：

 - 麦克风
 - 相机
 - 屏幕录制
 - 自动化（Automation）
 - 辅助功能（Accessibility）
 - 完全磁盘访问（如业务确需）
 - 联系人 / 日历 / 提醒事项 / 照片 / 文件访问等

 流程如下：

 1. Agent 检测任务依赖
 2. 发现权限不足则停止自动执行
 3. 生成待人工操作说明
 4. 人工进入系统设置完成授权
 5. Agent 复测并继续任务

 ### 4.3 标准提示模板

 建议统一输出：

 │ 当前任务需要 macOS 隐私权限。
 │ OpenClaw 不会自行修改 TCC 设置，也不会绕过系统授权。
 │ 请由人工在“系统设置 → 隐私与安全性”中完成授权后，再重新执行该任务。

 - credential 隔离

 附录 A：推荐实施边界

 自动执行

 满足以下条件可自动执行：

 - 不需要 sudo
 - 不需要新增 TCC 授权
 - 不跨用户
 - 不改系统级设置

 满足以下情况必须先确认：

 - 需要 sudo
 - 改 LaunchAgent
 - 改基线
 - 改 cron
 - 安装新组件
 - 调整安全策略

 仅提交人工操作

 以下内容不自动执行，只提交说明：

 - TCC 授权
 - sudoers 配置
 - 管理员身份变更
 - 系统级隐私配置
 - 跨用户访问授权
 - 高风险持久化策略变更

 附录 B：本修订版与原高权限版的核心差异

 本修订版相较于“Root 权限最大化能力模型”，核心变化如下：

 1. 运行模型：从 root / 高权限主导，改为普通用户运行
 2. 提权模型：从默认可控系统层，改为有限 sudo 白名单
 3. 隐私模型：从可讨论系统级自动化，改为 TCC 人工确认
 4. 多用户模型：从统一能力面，改为默认不互信隔离
 5. 落地策略：从“能力最大化”改为“边界清晰、逐步放权”
 6. 审计目标：从全系统控制，改为“当前 OpenClaw 版本在 macOS 上稳定、可解释、可回滚地运行”
